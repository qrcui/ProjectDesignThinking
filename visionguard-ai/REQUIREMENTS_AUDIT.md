# VisionGuard AI 需求符合性审计

审计日期：2026-07-17  
需求来源：`Team_Assignment_2.pptx`、`Team_Assignment_3.pptx`、`Team_Assignment_4.pptx`

## 修改前未完成项

本轮检查开始时，项目已有浏览器端摄像头监测、距离校准、视力测试和本地历史，但下列 PPT 需求尚未完整落地：

1. 缺少明确的 Access → Consent → Calibrate → Screen → Explain → Guide 六阶段引导和全程计时。
2. 摄像头启动前没有独立、可撤回的知情同意；没有分别说明原始影像与衍生结果的处理方式。
3. 缺少环境光检查、40–80 cm 支持范围，以及筛查前症状输入。
4. 结果没有按 Normal / Caution / Concern 分层，也没有“结果、主要指标、可信度与限制、下一步”四部分解释。
5. 摄像头流程没有在所有相关界面提供完整的暂停、停止、退出组合，也没有不中断流程的手动降级路径。
6. 缺少复测提醒、纵向趋势、可分享摘要、校园活动入口、二维码和转介入口。
7. 只有英文，缺少简体中文、繁体中文、英文选择和“默认简体中文”。
8. 若干实现细节会削弱结果可信度：Landolt C 几何比例、最差视力符号、无脸时长累计、跨会话距离校准、按键连发、演示数据建议等。

## PPT 量化指标核对

“已实现”表示代码路径存在且自动化检查通过；“需实测”表示不能仅凭代码诚实宣称达到精度或现场目标。

| 指标 | 代码/自动化证据 | 结论 |
|---|---|---|
| 完成一次筛查 ≤ 5 分钟 | 六阶段计时、简化流程、5 分钟目标提示 | 功能已实现；仍需可用性实测 |
| 视力结果与标准表相差 ≤ 1 行 | 物理屏幕校准、自适应 Landolt C、修正后的标准几何 | 需真人与标准视力表对照验证 |
| 眨眼检测 F1 > 0.85 | MediaPipe 眼部特征与眨眼状态机 | 需带标注视频集测量 Precision / Recall / F1 |
| 40–80 cm 内距离误差 ≤ 5 cm | 会话级人脸尺度校准与 40–80 cm 范围约束 | 需用卷尺在多设备、多距离点标定验证 |
| 每个结果包含 4/4 解释要素 | 端到端检查找到恰好 4 个解释区块 | 已验证 |
| 指导反馈 ≤ 2 秒 | 分析节流约 110 ms，结果保存后立即解释 | 代码满足；低性能目标设备仍需计时 |
| 异常结果给出匹配建议 | 风险单元测试覆盖视力、疲劳、组合信号和红旗症状 | 已验证 |
| 每个结果显示免责与专业照护提示 | 结果组件固定渲染，不依赖风险等级 | 已验证 |
| 保存原始图片/视频数量 = 0 | 无帧持久化或上传代码路径；仅可选保存结构化结果 | 已验证（架构） |
| 摄像头界面控制覆盖率 = 100% | 预览/校准区与测试区均提供控制，暂停会停用媒体轨道 | 已验证（界面） |
| 校园管理员部署 ≤ 10 分钟 | 静态构建、单命令本地服务器、二维码活动卡 | 需在目标校园环境计时验收 |
| 预约、专用设备、工作人员 = 0 | 自助浏览器流程、摄像头或手动降级 | 已实现 |

## 仍需外部完成的证据

项目功能层面的缺口已经补齐，但以下内容不能用前端代码代替：

- 使用真人、标准视力表和卷尺验证视力、眨眼 F1、距离误差三项精度指标。
- 在目标设备上验证 5 分钟完成率和 ≤2 秒反馈，并在校园网络上记录一次 ≤10 分钟部署验收。
- 部署方提供真实校园名称、健康中心/眼科转介链接与负责人；未配置时界面明确显示为部署占位信息。
- Assignment 2 中访谈、观察或用户反馈的研究证据仍应由课程团队在交付材料中提供，不能由产品界面虚构。

因此，当前结论是：**PPT 中可由产品代码实现的功能需求已覆盖；涉及临床/测量精度、真实用户研究和校园现场部署的数据性要求仍需实证后才能标记为“完全验证”。**
