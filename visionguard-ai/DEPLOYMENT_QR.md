# 二维码随扫随用部署方案

VisionGuard AI 的二维码应指向一个稳定的 **HTTPS 网页地址**，而不是把程序文件编码进二维码。用户扫码后由手机浏览器加载应用；摄像头分析仍在本机完成，当前版本不需要账号、数据库或应用服务器。

## 推荐路径：静态 HTTPS 托管

这是当前项目成本最低、最适合展示和校园试点的路径。

1. 在 `visionguard-ai` 目录生成发布文件：

   ```bash
   npm ci
   npm run build
   ```

2. 将 `dist/` 发布到 Cloudflare Pages、GitHub Pages、Vercel，或学校已有的静态网站空间。
3. 确认最终地址使用 HTTPS，并在 Android 与 iPhone 上分别测试摄像头授权。
4. 用最终地址打开应用。页面内“校园访问”区域会为当前完整地址生成二维码，可复制链接或打印卡片。

本项目已将 Vite 的生产资源路径设为相对路径，因此 `dist/` 可以放在域名根目录，也可以放在类似 `/visionguard/` 的子目录。若使用 GitHub Pages 的专用构建工作流，也可以按实际仓库名显式设置 Vite `base`。

建议二维码使用短而稳定的地址，例如：

```text
https://eye.example.edu/start
```

可以让该地址重定向到带活动参数的实际页面。以后更换托管平台时只修改重定向，不必重新印刷二维码。活动参数只能放公开信息，不能放检查结果或个人健康信息：

```text
https://eye.example.edu/?campus=North%20Campus&campaign=Healthy%20Eyes%20Week
```

## 其他技术路径

### 1. 临时演示：HTTPS 隧道

先在电脑启动发布版：

```bash
python serve_dist.py --no-browser
```

再用 Cloudflare Tunnel 等工具把 `http://127.0.0.1:4173` 暂时映射为公网 HTTPS 地址，并为该临时地址生成二维码。这适合当天演示，不适合正式上线：电脑必须一直开机，临时域名可能变化，并且需要先确认学校的网络和数据政策。

单纯把二维码做成 `http://127.0.0.1:4173` 无法让手机访问，因为手机的 `127.0.0.1` 指向手机自己。把服务绑定到局域网 IP 虽能解决寻址，但普通 `http://192.168...` 通常不属于浏览器摄像头允许的安全上下文，因此仍应配置受手机信任的 HTTPS。

### 2. 校园内网部署

把 `dist/` 放到学校的 Nginx、IIS、对象存储静态站点或门户子目录，配置校园 DNS、受信任的 TLS 证书和 `Permissions-Policy: camera=(self)`。手机需要连接校园 Wi-Fi 或 VPN。此路径便于由学校控制域名、日志和版本，但证书、网络连通性和隐私告知要由校方运维确认。

### 3. PWA（扫码后可“添加到主屏幕”）

在静态托管基础上增加 Web App Manifest、192/512 图标和 Service Worker，可让支持的手机浏览器把网页安装到主屏幕，并缓存界面、模型和 WASM 资源。PWA 改善重复使用和弱网体验，但**不会**让网页获得真正的后台摄像头能力；页面隐藏或系统冻结后，摄像头监测仍会被中断。

### 4. 原生/混合移动应用

若未来必须使用设备级 AR/深度能力、企业分发、推送提醒或更深的系统集成，可用 Capacitor 包装现有 React 前端，再分别实现 Android/iOS 原生插件。二维码可做成 Universal Link/App Link：已安装时打开 App，未安装时回退到网页。开发、签名、商店审核和隐私合规成本明显更高，而且后台使用摄像头仍受操作系统限制，不能设计成无提示的持续采集。

### 5. 带后台服务的产品化路径

只有在需要跨设备账户、集中报告、辅导员工作台或远程转诊时才增加 API、数据库和身份认证。届时需单独设计明确同意、最小化上传、访问控制、保留期限、删除机制和安全审计。当前本地优先版本不应为了二维码访问而上传任何报告。

## 建议实施顺序

1. **试用版**：静态 HTTPS 托管 + 稳定短域名 + 当前页面内二维码。
2. **复用版**：增加 PWA 安装与离线资源缓存。
3. **机构版**：确有集中管理需求后再增加受控后台。
4. **设备能力版**：只有网页能力明确不足时再做 Capacitor/原生应用。

上线前至少验证：HTTPS、首次摄像头权限、拒绝权限后的回退、iOS/Android 纵向与横向、二维码纠错与打印尺寸、模型首载时间、隐私文案、本地数据删除，以及校园网络下的实际可达性。
